# Aethium — Architecture

Binding technical decisions for the Aethium framework. Alternatives are listed only to document rejection.

---

## Rendering Strategy Decision

**Primary approach: (a) Immediate-mode canvas rendering via Wasm (browser) and the same immediate-mode command stream on native desktop (see Desktop Target Strategy).**

The developer describes UI each frame (or on dirty regions) as a retained **scene graph in Go memory**, but the **output** is an immediate-mode stream of draw commands (`FillRect`, `DrawText`, `Clip`, etc.) executed by a thin platform backend. There is no browser DOM tree and no native widget tree for application UI.

### Rejected: (b) Retained-mode DOM manipulation via `syscall/js`

| Reason | Detail |
|--------|--------|
| Binary size | `syscall/js` pulls a large JS interop surface; every DOM op crosses the Wasm boundary. |
| Performance | DOM layout thrashing and browser reflow are hostile to ≤ 150 ms startup and steady 60 fps on mid-range hardware. |
| Consistency | Desktop path would either duplicate DOM in WebView (two rendering models) or abandon DOM on desktop—already forcing a second implementation. |
| Budget risk | Meeting ≤ 500 KB gzipped with DOM frameworks + Go Wasm interop is historically unrealistic without TinyGo and still fragile. |

### Rejected: (c) Native OS widget bindings via CGo as the **primary** model

| Reason | Detail |
|--------|--------|
| Dual implementations | Win32 UIKit/AppKit/GTK would triple platform code; violates anti-bloat engineering cost. |
| CGo cost | CGo disables some optimizations, increases build complexity, and breaks pure Wasm builds. |
| Look-and-feel vs size | Native widgets optimize OS integration, not minimal binary or identical web/desktop visuals. |

**Note:** CGo may be used **only** inside the desktop host shell (WebView loader), not for per-component widgets.

## Wasm Toolchain Decision

| Target | Toolchain | Version targets |
|--------|-----------|-----------------|
| Browser / Wasm | **TinyGo** | TinyGo **0.34.0**; LLVM 18 backend bundled with that release |
| Native desktop / CLI | **Official Go toolchain** | Go **1.22.7** minimum (1.22 LTS line); `go.mod` declares `go 1.22` |

### Rejected: Standard `GOARCH=wasm GOOS=js`

A minimal `package main` with `fmt` commonly produces **2–3 MB+** Wasm before gzip—already 4–6× the **500 KB gzipped** exit criterion. Debuggability is better, but the size budget is non-negotiable for the browser path.

### Rejected: Custom runtime bridge (e.g. proprietary Wasm host embedding V8)

Introduces a third runtime to audit and ship—contradicts anti-bloat and AGPL source clarity. No proprietary bridge.

### Justification for TinyGo (browser)

- **Binary size:** TinyGo Wasm builds routinely land **150–400 KB** gzipped for non-trivial apps when stdlib use is restricted—within budget headroom.
- **Stdlib support:** Aethium core uses a **TinyGo allowlist** subset (see `BUILD_SYSTEM.md`); unsupported packages fail at CI, not at user laptops.
- **Debuggability:** TinyGo supports `-tags debug` with DWARF where enabled; browser path uses source maps generated by `aethium dev` mapping TinyGo locations to Go source. Native desktop builds use full Delve debugging on the Go toolchain path.

**Policy:** Application authors write **one** Go codebase; `aethium build` selects TinyGo vs `go build` per target.

## Concurrency Model

### Constraint

WebAssembly in browsers is **single-threaded** for Wasm linear memory access. The OS UI thread on desktop is likewise **single-threaded** for command submission to the canvas backend.

### Mapping goroutines

| Goroutine class | Runs on | Purpose |
|-----------------|---------|---------|
| **UI goroutine** (`runtime.UI`) | Main thread only | Frame tick, signal flushes, scene build, draw list submit |
| **Worker goroutines** (`runtime.Worker`) | Background | HTTP, file I/O, decode, compute |
| **Bridge callbacks** | Main thread | Results from workers delivered via scheduled callbacks |

**Rule:** Only the UI goroutine may call `Canvas`, `Signal` write paths that touch the scene, or lifecycle hooks.

### Scheduling without blocking rendering

1. Workers send results on `chan T` buffered (default capacity **64**).
2. The UI goroutine registers interest via `runtime.ScheduleOnUI(fn func())`, which enqueues `fn` on an internal **UI queue** drained once per frame before paint.
3. On Wasm, `ScheduleOnUI` is implemented with `syscall/js` scheduling a host callback (`queueMicrotask` + `requestAnimationFrame` coalescing) so worker completions never call Go from arbitrary JS threads— they **only** append to the UI queue from the host’s designated entry stub.
4. On desktop, the WebView/GL host’s event loop invokes `aethium_runtime.PumpEvents()` which drains the UI queue—same API, different host.

### Channel / callback boundary

```
[Worker goroutine] --chan--> uiMailbox
                                |
                                v
[Host event loop] --PumpEvents--> drain uiMailbox --> ScheduleOnUI closures
                                |
                                v
[UI goroutine / same thread] --> ApplySignals --> BuildFrame --> SubmitDrawList
```

**Forbidden:** Blocking `<-chan` on the UI thread inside `Mount`/`Update`/`Paint`.

## Component Lifecycle

Lifecycle is identical on Wasm and desktop; only the canvas backend differs.

### Stages

| Stage | Trigger | Go construct | Responsibility |
|-------|---------|--------------|----------------|
| **Init** | `NewComponent(props)` | `Component` interface + `Props` struct | Allocate pooled `VNode`, register signals, **no I/O** |
| **Mount** | First inclusion in active tree | `MountContext` | Subscribe signals, async kickoff via `runtime.Schedule`, first `Build` |
| **Update** | Signal change or parent re-render | `UpdateContext` | `Build` only if subscribed signals dirty |
| **Unmount** | Removed from tree | `UnmountContext` | Unsubscribe, return `VNode` to `sync.Pool`, cancel `context.Context` |

### Core interfaces (binding names)

```go
// Package: github.com/A-Solo-Engineer/aethium/runtime

type Component interface {
    Init(ctx InitContext) error
    Mount(ctx MountContext) error
    Update(ctx UpdateContext) error
    Unmount(ctx UnmountContext) error
}

type InitContext struct {
    Props   any
    Runtime *Runtime
}

type MountContext struct {
    Index int
    Node  *scene.VNode
}

type UpdateContext struct {
    Index int
    Dirty []reactive.SignalID
    Node  *scene.VNode
}

type UnmountContext struct {
    Index int
    Node  *scene.VNode
}
```

### Runtime orchestration

- `Runtime.Attach(root Component) error` — builds tree, runs Init→Mount depth-first.
- `Runtime.Tick() error` — drains UI queue, flushes signals, runs Update on dirty subtrees, emits draw list.
- `Runtime.Detach(root scene.NodeID) error` — Unmount subtree.

**Hand-off to state:** Signals are defined in `STATE_MANAGEMENT.md` (`Signal[T]`, `SignalID`). Components read signals in `Build` (via `View` functions returning `DrawCmd` slices)—not in Init.

## Desktop Target Strategy

**Decision: Bundled system WebView as the desktop host shell** (Windows: WebView2 via `github.com/webview/webview` + loader; macOS: WKWebView; Linux: WebKitGTK via same webview abstraction).

The desktop binary ships:

1. **Native Go binary** (≤ 5 MB budget includes loader + Wasm runtime glue OR embedded precompiled Wasm loaded into the WebView—see `BUILD_SYSTEM.md`).
2. **WebView** displaying a minimal `index.html` with WebGL2 canvas.
3. **Same TinyGo-produced Wasm module** as the browser target, loaded from `embed` or sidecar.

### Rejected: CGo bindings to platform UI toolkits

Fragmented APIs, accessibility burden per toolkit, and two rendering models (widgets vs canvas). Fails the “one codebase” principle.

### Rejected: Custom OpenGL/Metal/Vulkan surface as primary (non-WebView)

Highest long-term performance potential but Stage 2 cost explodes (GPU shaders, input, text shaping per platform). Deferred; if pursued later, it must consume the **same** `DrawList` IR from immediate-mode—never a second component API.

### Why WebView is acceptable under anti-bloat

Aethium does **not** ship Chromium. It uses the **OS-installed** WebView2/WebKit (tens of MB shared system libraries, not 120 MB bundled Electron). The **application binary** stays within the 5 MB exit criterion; system WebView RAM is excluded from the app binary metric but included in idle RAM measurements with documented methodology in `BUILD_SYSTEM.md`.

## Implementation Deviations from Stage 1 Spec

| Decision | Stage 1 Spec | Stage 2 Reality | Reason |
|---|---|---|---|
| Context embedding | `MountContext` embedded `InitContext` | Flat structs with `Index` and `Node` fields | Simpler pool-friendliness; embedding caused field shadowing |
| `Runtime.Tick` signature | `Tick(frame FrameInfo)` | `Tick() error` | `FrameInfo` deferred; frame counter managed internally via `GetFrame()` |
| VNode pool | Described in STATE_MANAGEMENT.md | Implemented in `scene/scene.go` as `vnodePool sync.Pool` | Correctly placed; pool is scene-layer concern |
